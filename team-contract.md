## Project Start
This project started on Monday, February 9, 2026.

## Communication
* We will use GitHub Issues to track features and bugs.
* We will use Slack for daily chat.
* We try to reply within 24 hours.

## Working Style and Progress Updates
* Before each weekly meeting, each member will post a short progress update in Slack (what they did, what they will do next, and any blockers)
* During the weekly meeting, we will briefly review these updates and adjust tasks or priorities if needed.

## Meetings
* We will meet weekly on Thursdays at 6 PM PST.
* We will rotate the responsibility for creating the meeting agenda and taking minutes each week.
* If someone cannot make it, please let the team know early.

## Work Process and Fairness
* We will break the project into tasks and assign them fairly based on team member skills and availability.
* We expect high-quality work and will review each other's code to ensure it meets our standards.

## Work Hours
* We expect to work on this project for about 5-8 hours per week each.
* We will try to avoid working on Sundays due to family obligations.
* In addition to avoiding Sundays, members should inform the team at least one week in advance about any periods of limited availability.

## Git Workflow
* All code changes must be done on a new branch, not directly to main.
* Before merging code, we must have a code review.
* We must write clear commit messages.

## Quality Expectations
* Code should run without errors, include basic comments or docstrings where appropriate, and follow a consistent style agreed by the team.
* Each pull request should include a short description of the changes and, when relevant, screenshots or notes on how to test the feature.

## Rules
* We must follow the project [Code of Conduct](CODE_OF_CONDUCT.md).
* Be respectful and constructive.
* Harassment or discrimination is not allowed.

## Conflict Resolution
1. Talk directly to the team member.
2. If it is not solved, bring it to the team meeting.
3. If it is still not solved, ask the TA for help.

## Decision Making and Contract Changes
* We will aim for consensus when making important decisions about the project scope and design. If we cannot reach consensus, we will make decisions by majority vote.
* This team contract may only be changed if all team members agree to the changes.
